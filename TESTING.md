# Инструкция по тестированию (лаба 3 — P2P чат)

Эта инструкция составлена по разделу «Способы тестирования чата в многопользовательском режиме» из `P2P чат.pdf`.

## Вариант A (рекомендуется для защиты): macOS + Windows VM в одной сети

Этот вариант соответствует пункту из PDF про тестирование на «две и более реальные машины / VM», только в роли второй машины — Windows VM.

### 0) Подготовка Windows VM (один раз)

В Windows нужен Python 3 (macOS Python нельзя использовать из Windows).

Самый простой способ через PowerShell:

    winget install -e --id Python.Python.3.12

Если после установки `python` всё равно ведёт в WindowsApps, отключите алиасы:
Settings → Apps → App execution aliases → выключить `python.exe` / `python3.exe`.

Проверка:

    python --version

### 1) Сетевая настройка Parallels

Чтобы UDP broadcast discovery работал стабильнее, предпочтительно:
- **Bridged** (VM как отдельное устройство в вашей Wi‑Fi/LAN сети).

В режиме NAT/Shared UDP broadcast может не доходить.

### 2) Узнать IP-адреса (НЕ 127.x.x.x)

На macOS (Wi‑Fi интерфейс обычно `en0`):

    ipconfig getifaddr en0         192.168.1.8

На Windows:

    ipconfig     IPv4 Address. . . . . . . . . . . : 192.168.1.11

Берите IPv4-адреса вида `192.168.x.x`.

### 3) Запуск узла на macOS

В терминале macOS:

    cd "/Users/volkov_sh/Desktop/Ксис лабы/лаб3"
    python3 p2p_chat.py --name Mac --bind-ip <IP_Mac> --udp-port 50000 --tcp-port 50001

### 4) Запуск узла на Windows VM

В CMD/PowerShell Windows:

    cd /d "Z:\Desktop\Ксис лабы\лаб3"
    python p2p_chat.py --name Win --bind-ip <IP_Windows_VM> --udp-port 50000 --tcp-port 50002

### 5) Что должно быть видно при успешном тесте

В логах (с отметками времени):
- «Обнаружен узел …» (получен UDP discover);
- «TCP-соединение установлено …» (выполнен connect + обмен hello/именем).

Далее:
- отправьте текст на macOS → он должен появиться на Windows (входящее сообщение с именем и IP отправителя);
- отправьте текст на Windows → он должен появиться на macOS;
- проверьте `/history` на обоих узлах;
- завершите один узел командой `/quit` — второй должен увидеть событие отключения (UDP leave и/или закрытие TCP).

### 6) Если узлы не видят друг друга (чек-лист)

1. Проверьте, что VM действительно в Bridged и имеет IP в той же подсети, что macOS.
2. Проверьте firewall Windows: разрешить входящий **UDP 50000** и входящий **TCP 50002**.
3. Для диагностики на macOS можно посмотреть, приходят ли UDP discovery:

    sudo tcpdump -ni any udp port 50000

## Вариант B: Docker (для консольных программ)

Идея: запустить несколько контейнеров в одной Docker-сети. Каждый контейнер будет отдельным «компьютером».

Общий план:
- собрать образ с Python;
- поднять 2–3 контейнера в одном bridge network;
- каждому задать своё имя и свои порты.

(Этот вариант зависит от вашего окружения и не обязателен, если вариант A работает.)

## Вариант C: две и более реальные машины / VM

Идея: запустить программу на 2+ компьютерах в одной локальной сети.

Шаги:
1. На каждом ПК выбрать:
- свой IP (например, `192.168.1.10`, `192.168.1.11`);
- общий UDP port discovery (например, 50000);
- свой TCP port (например, 50001, 50002).

2. Запустить на каждом:

    python3 p2p_chat.py --name <имя> --bind-ip <IP_машины> --udp-port 50000 --tcp-port <TCP_порт>

3. Узлы должны обнаружить друг друга по UDP и соединиться по TCP.

4. Если узлы не видят друг друга:
- проверьте, что они в одной подсети;
- проверьте firewall (разрешить UDP 50000 и TCP порты);
- проверьте, что broadcast пакеты не фильтруются.

## Примечание про loopback (127.0.0.x)

В PDF предлагается тестировать через bind на `127.0.0.1`, `127.0.0.2`, …, но на macOS/Windows в связке с VM UDP broadcast discovery по loopback может не работать стабильно. Для демонстрации между macOS и Windows VM используйте вариант A (реальные IP в LAN + Bridged).
